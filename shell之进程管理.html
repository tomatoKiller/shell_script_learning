shell之进程管理

shell 之进程管理

利用shell进行一些和linux进程相关的处理动作，是要借助于一些命令的。下面简单介绍一下常用的命令。

谈到进程相关的命令，首当其冲就是ps命令。

ps命令是Process Status的缩写，主要用来显示当前系统中正在运行的进程状态。

常用选项

  • -A 显示所有进程
  • -e 显示所有进程和A选项相同
  • -m 进程后面紧跟着显示线程信息
  • -p 后跟pid作为输入
  • -f 显示UID,PPIP,C与STIME栏位
  • -L 显示线程信息，和-f选项一起用可以显示LWP和NLWP列
  • -o 指定显示的格式

常用的选项组合

ps命令常用的2种选项组合分别是

ps aux      # BSD风格。选项前面可以不加-
ps -elf     # standard风格。选项前面必须加-

下面是两个风格的选项组合输出的内容

# ps aux
USER PID %CPU %MEM VSZ RSS TTY STAT START TIME COMMAND
# ps -elf
F S UID PID PPID C PRI NI ADDR SZ WCHAN STIME TTY TIME CMD

字段含义解析

字段 含义
USER 进程启动用户
UUID 即EUID, 有效用户ID
PID 进程ID
PPID 父进程ID
%CPU/C 进程占用的CPU百分比
%MEM 进程占用的内存的百分比
F 进程标志
PRI 进程的优先级，值越大，优先级越低
NI 进程的NICE值，表示进程可被执行的优先级的修正数值
VSZ 进程的虚拟内存大小，即使用的地址空间大小，单位KB
RSS 进程实际占用的内存大小，不包括被内核交换出去的代码，但是包含了动态库的内存。单位KB
TTY 进程终端ID
STAT/S 进程状态
WCHAN 进程被挂起的内核函数的地址
START/STIME 进程启动时间
TIME 进程消耗的CPU时间
COMMAND/CMD 进程的启动命令

进程状态含义

状态 含义
D 不可中断休眠(一般是IO操作)
R 运行中或者可运行（在就绪队列中）
S 可中断休眠，一般是在等待事件完成
T 任务暂停。可能接收到了控制信号而暂停，或者是正在被跟踪调试
X 进程死亡(这个状态应该永远也看不到)
Z 僵死进程

日常工作及问题定位中，需要查看进程以及其子进程、线程之间的关系，可以通过-eLf选项来实现，该命令的结果中，

进程管理

进程执行方式

在shell中，支持两种进程执行方式：顺序执行和并发执行。

# 下面四个命令依次顺序执行
cmdA
cmdB
cmdC; cmdD

# 后台执行cmdA 然后前台执行cmdB
cmdA & cmdB

&和;号分割的命令组合执行完成后，返回的是最后一条命令的退出状态。

在shell中，由&号触发进程在后台执行后当前shell无需等待进程执行结束而会立即返回(返回值为0)。如果没有使用job控制功能且没有明确进行重定向时，被放在后台执行的进程其标准输入被重定向为/dev/null。

job control

shell提供了job control功能，通过这个功能可以有选择的来暂停某个进程的执行，或者将某个被暂停的进程恢复执行。这个功能很强大。一般来讲这个功能在shell的交互界面中用的多一些。

job control操作的对象是job而非某个单一的命令。如果多个命令通过管道符号连接起来一起提交的话则视为一个job。shell内部会维护一个当前正在执行的job的列表。

当异步启动一个job时，shell会打印出jobid和进程id。通过jobs命令可查看当前的job及其状态。

>$ sleep 100 &
[1] 21977           #jobid=1  processid=21977
>$ sleep 200 &
[2] 22004           #jobid=2  processid=22004
>$ jobs
[1]  - running    sleep 100     # -号表示 倒数第二个提交的job
[2]  + running    sleep 200     # +号表示 最新(倒数第一个)提交的job

前台执行的任务可以通过 Ctrl+Z 按键来将其放到后台(此时任务会暂停执行)，进而把控制全返回给shell。 fg命令可以将后台任务放到前台来执行。 bg命令则可以将后台目前暂停执行的任务继续开始执行。

在使用fg和bg命令时，可以指定job的id。

bg %5       # 后台执行jobid=5的job
%5 &        # 含义同上
fg %5       # 前台执行jobid=5的job
%5          # 含义同上
bg %%       # 后台执行最近提交的job
bg %+       # 含义同上
bg %        # 含义同上
bg %-       # 后台执行倒数第二个提交的job

除了通过jobid来指定job之外，还可以使用启动job的命令名前缀或通配符来指代job，但如果有多个job符合条件，则报错。

>$ sleep 200
^Z
[2]  + 29216 suspended  sleep 200
>$ bg %sle  # 后台执行sle开头的命令
[2]  - 29216 continued  sleep 200
>$ fg %?le  # 后台执行包含le字符串的命令
[2]  - 29216 running    sleep 200

>$ sleep 100 &
[1] 29693
>$ sleep 200 &
[2] 29697
>$ fg %?le
bash: fg: le: ambiguous job spec

coproc命令启动后台进程

Bash从版本4开始提供了一个coproc命令，它可以在后台启动进程来执行命令，类似于使用了后台命令控制符"&"，不同的是coproc会在其启动的后台进程会和当前shell进程之间创建一个双向管道，方便父子进程之间的通信。coproc命令的语法如下面的伪代码所示：

coproc [NAME] command [redirections]

上面示例中的NAME是创建的进程名称，如果不指定的话则默认是COPROC。如果command是一个简单命令的话，则不能指定NAME，否则shell会把NAME当成是command来执行，进而报错，针对简单命令，可以把命令放在{}中，这个时候就可以指定NAME了。执行coproc命令之后，父子进程之间的双向管道存储在一个数组变量中，数组变量名称就是指定的NAME。NAME[0]是后台进程的标准输出，NAME[1]是后台进程的标准输入。生成的后台进程的进程ID存储在变量NAME_PID中。

{ coproc mycoproc { echo "begin_"$0"_end"; } >&3; } 3>&1

echo $mycoproc_PID
echo ${mycoproc[0]}
echo ${mycoproc[1]}
echo abc> ${mycoproc[1]}

nohup 与 disown

上面讲了，在shell中可以执行一些后台命令。但是默认情况下，在当前shell收到SIGHUP信号(比如ssh登录的pty断开时)时，它也会向自己启动的所有后台进程发送SIGHUP，而该信号的默认处理行为是退出进程。因此当前shell挂断时，其之前启动的所有后台进程都会退出。如何阻止这种情况发生呢，这里就要用到nohup或disown命令了。

注意：这两个命令往往是跟后台命令(&)搭配使用

当使用nohup来执行后台命令时，会产生下面几种影响：

  • 关闭标准输入，nohup执行的命令无法读取标准输入
  • 将标准输出和标准错误重定向到nohup.out文件
  • nohup自动的进程不再接受SIGHUP信号

当使用disown来执行后台命令时，会产生下面几种影响：

  • 把job从当前shell的job列表中移除。因此通过jobs指令就看不到了，也就接收不到当前shell发送的SIGHUP信号。如果使用disown -h来操作job时，则并不会真正将job从shell的job列表中移除，而只是令程序不在接受SIGHUB信号了。
  • 标准输入和标准输出仍然和终端相连，因此一旦终端退出，则程序无法再读取标准输入或者写标准输出
# nohup 必须放在所要执行的命令前面
nohup sleep 100 &

# disown 要放在所要执行的命令后面，相当于使用&号来串联前面的指令和最后的disown指令。直接不带参数使用disown指令表示处理的是最近提交的job
sleep 100 & disown

# 也可以使用disown命令直接操作job也可以
bash-3.2$ sleep 100 &
[1] 38870
bash-3.2$ jobs
[1]+  Running                 sleep 100 &
bash-3.2$ disown %1
bash-3.2$ jobs
bash-3.2$

nohup多条指令在后台顺序执行。

工作中有时候会遇到一下需求

  1. 后台执行多条指令
  2. 多条指令必须顺序执行
  3. shell退出后不能结束进程
nohup bash -c 'cmd1;cmd2;cmd3' > file &
# 或者
{ cmd1;cmd2;cmd3;} > file & disown
